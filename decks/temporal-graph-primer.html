Temporal Graph가 정확히 뭐냐

한 줄로 말하면, 시간이 지나며 바뀌는 사실을 삭제하지 않고 언제 참이었는지, 지금도 참인지, 무슨 근거로 아는지까지 저장하는 그래프다.

mental model

일반 그래프는 관계를 저장하고, temporal graph는 관계의 수명까지 저장한다

Node

Entity

사람, 프로젝트, 회사, 도구, 취향 같은 대상. 예: User, Puma, rem, SQLite.

Edge

Fact

두 대상 사이의 주장. 예: User PREFERS Puma, rem USES SQLite.

Time

Validity

그 주장이 언제부터 언제까지 참이었는지. 현재 참이면 종료 시각이 비어 있다.

example

"아디다스 좋아함"과 "이제 푸마가 좋음"은 충돌이 아니라 시간 변화일 수 있다

기존 메모리 시스템은 둘 다 검색해서 모델을 헷갈리게 만들 수 있다. Temporal graph는 옛 사실의 창을 닫고 새 사실을 현재 사실로 둔다.

the difference

flat memory vs temporal graph

Flat memory

  • 2025-01-10: User likes Adidas.
  • 2025-04-02: User now prefers Puma.
  • 검색 결과에 둘 다 나오면 현재 선호를 판단하기 어렵다.
  • 오래된 사실을 삭제하면 과거 맥락도 사라진다.

Temporal graph

  • Adidas fact: 2025-01-10부터 2025-04-02까지 valid.
  • Puma fact: 2025-04-02부터 현재까지 valid.
  • 현재 질문에는 Puma만 답한다.
  • 과거 질문에는 Adidas도 살아 있다.
timestamps

중요한 건 시간도 한 종류가 아니라는 점

valid time

세상에서 언제 참이었나

사용자가 실제로 푸마를 선호하게 된 날짜. Zep/Graphiti 문맥에서는 `valid_at`, `invalid_at`에 가깝다.

transaction time

시스템이 언제 알았나

`rem`이 그 사실을 기록하거나 커밋한 시각. `created_at`, `learned_at`, `expired_at`에 가깝다.

둘을 분리하면 "4월 2일부터 참이었지만 4월 5일에 알았다" 같은 상황을 표현할 수 있다. 이게 bitemporal 모델의 핵심이다.

query behavior

Temporal graph가 가능하게 하는 질문

현재 상태

"지금 사용자가 선호하는 브랜드는?" -> `valid_to is null`인 현재 fact만 본다.

과거 상태

"2025년 1월에는 뭘 좋아했나?" -> 해당 시각에 열려 있던 fact를 본다.

변경 이유

"왜 바뀌었나?" -> fact에 연결된 source episode를 열어 근거를 보여준다.

충돌 검토

"새 fact가 기존 current fact를 닫아야 하나?" -> semantic review의 핵심 작업이 된다.

zep / graphiti lesson

Zep/Graphiti에서 배울 점은 "그래프 DB"보다 "시간 달린 fact ledger"다

Entities

텍스트에서 명사를 뽑고, 같은 대상을 dedupe/merge한다.

Facts

관계를 edge로 저장하고 `valid_at`, `invalid_at` 같은 시간을 붙인다.

Episodes

원본 대화/문서/메모를 보존해서 fact의 근거를 추적한다.

즉, 오래된 fact를 없애는 게 아니라 "언제까지 참이었는지" 닫는다. 이게 semantic conflict 처리의 중심 모델이 될 수 있다.

what rem should do first

`rem`은 외부 graph DB부터 붙이지 말고 SQLite에 temporal graph 모양의 cache를 먼저 만들자

semantic_entities(id, canonical_name, kind, summary) semantic_episodes(id, source_memory_id, created_at, text_hash, excerpt) semantic_facts( id, subject_id, relation, object_id_or_value, valid_from, valid_to, learned_at, expired_at, source_memory_id, confidence ) semantic_fact_sources(fact_id, episode_id)
architecture fit

현재 `rem` 구조에는 derived temporal index가 잘 맞는다

그대로 유지

  • Markdown이 canonical source.
  • Git commit이 transaction boundary.
  • SQLite는 rebuild 가능한 local cache.

추가할 것

  • entity/fact/episode derived tables.
  • fact validity windows.
  • semantic fact diff/review workflow.

Canonical 데이터를 바꾸지 않고 cache만 확장하면, 실패 시 rollback과 deterministic rebuild 원칙을 지킬 수 있다.

do not confuse these

Git conflict와 semantic conflict는 다른 층이다

Git conflict

같은 파일/라인을 여러 쪽에서 수정해서 Git merge가 자동으로 못 끝내는 상태. 파일 병합 문제다.

Semantic conflict

파일 병합은 됐지만 의미상 현재 사실이 충돌하는 상태. 예: 선호 브랜드, 직장, 프로젝트 상태가 바뀜.

Temporal graph는 두 번째 문제를 다룬다. "둘 중 하나 삭제"가 아니라 "각 fact의 유효 기간을 정리"하는 방향이다.

trade-offs

좋은 점과 위험한 점

장점

  • 현재 사실과 과거 사실을 분리한다.
  • 오래된 정보를 삭제하지 않고 무효화한다.
  • semantic conflict를 reviewable하게 만든다.
  • 모든 fact에 source/provenance를 붙일 수 있다.

비용

  • entity extraction과 merge가 틀릴 수 있다.
  • relation ontology가 없으면 edge가 난잡해진다.
  • LLM extraction은 deterministic rebuild와 충돌할 수 있다.
  • graph DB는 운영 복잡도를 크게 올린다.
bottom line

Temporal graph로 간다는 말의 현실적인 의미

`rem`에서 temporal graph는 "메모리를 그래프 DB로 갈아엎자"가 아니다. 더 정확히는, Markdown 원본 위에 시간 달린 entity/fact/episode index를 만들고, semantic conflict를 fact validity review 문제로 바꾸자는 뜻이다.

old approach: retrieve similar notes -> model guesses current truth temporal graph approach: retrieve current facts + source episodes + historical facts when needed
