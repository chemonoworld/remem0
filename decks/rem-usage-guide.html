rem 0.1.0 · 2026-07-10 KST

rem 사용법 전체 가이드

사람과 로컬 에이전트를 위한 Markdown memory CLI. Markdown은 원본이고, Git commit은 변경의 transaction boundary이며, SQLite는 재생성 가능한 검색·semantic cache입니다.

Canonical

Markdown + Git

메모리 파일과 Git history가 지속 상태입니다. SQLite DB는 Git에 들어가지 않습니다.

Required

GitHub / GitLab origin

모든 vault는 Git repository root여야 하고 지원되는 origin remote가 있어야 합니다.

Scope

활성 profile 기반

대부분의 명령은 현재 디렉터리가 아니라 config의 active profile vault를 사용합니다.

중요: rem doctor, rem facts, rem search에는 --root 옵션이 없습니다. 다른 vault를 대상으로 하려면 rem profile use <name> 또는 rem init --profile <name> --root ...로 활성 profile을 바꾸세요.

01 · local deployment

설치, 확인, 업데이트, 삭제

Repository에서 설치 스크립트를 실행하면 $HOME/.cargo/bin/rem에 user-level binary가 설치됩니다. 이 가이드는 설치된 rem 기준으로 표기합니다. 개발 중에는 앞에 cargo run --을 붙여 동일한 명령을 실행할 수 있습니다.

첫 설치

cd /path/to/remem0
./scripts/install-local.sh

cd /tmp
command -v rem
rem --help

스크립트는 cargo install --path . --locked --offline --force를 실행합니다. 필요한 Cargo dependency가 아직 local cache에 없다면 한 번 cargo build를 네트워크가 가능한 환경에서 실행하세요.

PATH, update, uninstall

# command -v rem 이 비어 있을 때
export PATH="$HOME/.cargo/bin:$PATH"

# repository 변경 또는 pull 후
git pull --ff-only
./scripts/install-local.sh

# 제거
cargo uninstall rem

zsh에서는 PATH export를 ~/.zshrc에 넣은 뒤 새 terminal을 엽니다.

02 · first vault

Vault 초기화와 profile

vault root는 subdirectory가 아니라 Git working tree의 최상위여야 합니다. origin은 GitHub 또는 GitLab host를 가리켜야 하며, HTTPS와 SSH remote 모두 사용할 수 있습니다.

mkdir -p ~/rem-vault
git -C ~/rem-vault init
git -C ~/rem-vault remote add origin git@github.com:you/rem-vault.git

rem init --root ~/rem-vault --storage git
rem profile list
rem doctor

Storage mode

local
일반 로컬 Markdown vault입니다. Git requirement는 동일합니다.
obsidian
Obsidian vault path를 root로 씁니다. Obsidian Sync는 optional이며 Git이 transaction log입니다.
git
Git-backed vault를 나타내는 profile mode입니다. 실제로는 모든 mode가 GitHub/GitLab origin을 요구합니다.

초기화 결과

<vault>/ memories/short/ memories/long/ policies/ inbox/ archive/ .rem/cache/ .rem/tx/

.rem/cache/와 .rem/tx/는 자동으로 .gitignore에 들어가며 commit에서 제외됩니다.

이미 dirty한 vault: 초기화 전에 변경이 있으면 interactive terminal에서 include / restore / abort를 고릅니다. script에서는 --accept-external 또는 --restore-external 중 하나를 명시해야 합니다.

여러 vault profile

rem profile add work ~/work-vault --storage git
rem profile add obsidian ~/Documents/MyVault --storage obsidian
rem profile list
rem profile show work
rem profile use work

03 · configuration

TUI와 low-level config

명령 없이 rem만 실행해도 configuration TUI가 열립니다. 명시적으로 열려면 rem configure를 사용합니다.

Interactive TUI

  • Up / Down 또는 j / k: field 이동
  • Enter: 선택 field 편집, 편집 중 Enter 적용 / Esc 취소
  • S: config 파일 기록. vault Git 조건은 초기화·mutation 시 검증합니다.
  • Ctrl+F: profile-root 선택 시 사용자 $HOME를 기본 root로 탐색하는 fuzzy directory finder 열기
  • finder: 처음에는 $HOME가 채워진 path filter에 focus됩니다. 그대로 수정하거나 Ctrl+A로 바꾼 뒤, Tab/Shift+Tab으로 path 목록 focus를 전환합니다. 목록 focus에서만 Up/Down 또는 j/k로 경로를 고릅니다.
  • I: 현재 profile vault를 초기화하고 transaction commit 생성
  • R: defaults로 reset (기록하려면 다시 S)
  • Q, Esc, Ctrl+C: 종료

Script-friendly config

rem config init
rem config path
rem config show

rem config set active-profile work
rem config set default-search bm25
rem config set editor "code --wait"

rem config reset

config key는 active-profile, default-search, editor뿐입니다. default search 값은 auto, grep, bm25, vector, all 중 하나입니다.

Config 위치: 기본은 $HOME/.rem/config.toml이고, CI·테스트·격리 환경에서는 REM_HOME=/tmp/rem-home rem ...로 global config 위치만 분리할 수 있습니다. vault root 자체는 profile이 결정합니다.

04 · memory lifecycle

메모리 추가, 조회, 편집, 보관, 승격

메모리 mutation은 Markdown 작성, strict SQLite reindex, Git commit을 한 transaction으로 처리합니다. ID 전체 또는 고유한 prefix를 <ID> 자리에 넣을 수 있습니다.

추가

rem add --short "임시 작업 문맥"

rem add --long \
  --scope project \
  --kind decision \
  --tag rust --tag cli \
  --source human \
  --agent codex \
  --session sprint-9 \
  "# Decision
Use Markdown as canonical memory."

type 기본값은 short, scope 기본값은 user, kind 기본값은 note, source 기본값은 cli입니다. --short, --long, --type short|long 중 하나로 type을 정합니다.

목록과 본문 조회

rem list
rem list --short --kind decision
rem list --type long --scope project --tag rust
rem list --all

rem show <ID-or-unique-prefix>

list filter: --short, --long, --type, --scope user|project|agent|session, --kind fact|preference|decision|task|procedure|note|question, --tag, --all. --all은 archive도 포함합니다.

수정

rem update <ID> "새 본문"
rem update <ID> --append "추가 문단"

rem edit <ID>

edit는 config editor, 없으면 $EDITOR, 없으면 vi로 파일을 연 뒤 종료 시 transaction을 수행합니다.

보관 또는 hard delete

# 기본: archive/로 이동
rem delete <ID>

# 파일을 제거
rem delete <ID> --hard

기본 delete는 status를 archived로 바꾸고 archive directory로 이동합니다. --hard는 Git history 외의 현재 파일을 제거합니다.

Short → long 승격

rem promote <short-ID>
rem promote <short-ID> "수정한 장기 메모리 본문"

short memory는 유지하고, 새 long memory를 만들며 새 파일의 promoted_from metadata에 원본 ID를 기록합니다.

모든 mutation 공통 옵션: --accept-external, --restore-external, --non-interactive. 대상은 init, add, edit, update, delete, promote입니다.

05 · Git transaction boundary

Manual Markdown 변경과 rem commit

Obsidian 또는 editor로 Markdown을 직접 바꾼 뒤에는 rem commit이 pending vault 변경을 검증하고, temporary SQLite index를 만든 뒤 atomic replace하고, cache/tx를 제외한 변경을 하나의 Git commit으로 기록합니다.

1. validateGit root, GitHub/GitLab origin, pending tx journal, external Git state를 확인합니다.
2. snapshotvault 파일, current index, Git staging 상태를 journal에 snapshot합니다.
3. applyCRUD 결과 또는 manual Markdown 변경을 transaction 범위에서 처리합니다.
4. reindextemporary SQLite에 rebuild하고 diagnostic이 없을 때만 index를 교체합니다.
5. commit변경된 vault path만 Git commit하고 journal을 정리합니다. 실패하면 snapshot으로 rollback합니다.

기본과 dry run

# editor / Obsidian에서 Markdown을 직접 변경한 뒤
rem commit

rem commit -m "sync manual vault edits"
rem commit --dry-run

--dry-run은 commit 대상과 reindex diagnostic을 출력하지만 Git commit은 만들지 않습니다. 기본 message는 rem: commit vault changes입니다.

Non-interactive script

# 외부 변경을 포함
rem commit --non-interactive \
  --accept-external \
  -m "sync agent changes"

# 외부 변경을 버리고 진행
rem commit --non-interactive \
  --restore-external

--accept-external과 --restore-external은 서로 배타적입니다. policy를 지정하지 않은 non-interactive 실행은 실패합니다.

Interactive external-change review

기본 prompt

dirty vault에서 mutation 또는 rem commit을 interactive terminal로 실행하면 commit / restore / abort를 선택합니다. 기본은 commit입니다.

상세 review

rem commit --review는 commit-all, pick, diff, restore-all, abort를 제공합니다. pick에서는 file마다 include, restore, diff, abort를 고릅니다.

Git merge conflict는 별도: Git status가 unmerged이면 --accept-external이나 review로 진행할 수 없습니다. 먼저 git status와 editor/Git tooling으로 conflict marker를 해결하고 git add 또는 git merge --continue를 끝낸 후 rem commit --review을 다시 실행하세요. Semantic conflict review는 아직 TODO이며 Git conflict review와 의도적으로 분리되어 있습니다.

06 · retrieval

Grep, BM25, auto 검색

검색 결과는 tab-separated id, memory_type, source, title, score, path입니다. default search mode는 config의 default-search이며, 명령에서 mode flag를 주면 그것이 우선합니다.

검색 명령

rem search "Markdown"
rem search --grep "exact local scan"
rem search --bm25 "SQLite"
rem search --all "decision"
rem search --mode auto "project context"

explicit mode option: --grep, --bm25, --vector, --all, --mode auto|grep|bm25|vector|all.

Mode 동작

grep
active Markdown memory의 title, body, tag를 case-insensitive하게 직접 scan합니다.
bm25
SQLite FTS5 index를 사용합니다. index가 없으면 rem rebuild가 필요합니다.
auto
grep와, 존재하면 BM25를 merge합니다. 기본값입니다.
all
현재 버전에서는 grep + 가능한 BM25 merge입니다.
vector
CLI value는 있으나 v1에서는 configured되지 않았으므로 오류를 반환합니다.

07 · temporal semantic cache

Temporal facts 작성과 조회

semantic graph는 Markdown body의 명시적 @fact directive에서 파생되어 SQLite cache에 저장됩니다. source Markdown이 원본이고 별도 graph database는 없습니다.

Markdown directive 문법

@fact subject | RELATION | object | key=value ...

@fact User | PREFERS | Adidas | valid_from=2025-01-10 | valid_to=2025-04-02
@fact User | PREFERS | Puma | valid_from=2025-04-02
@fact User | USES | LegacyTool | valid_from=2020-01-01 | expired_at=2024-01-01
@fact User | PART_OF | rem | object_kind=entity | confidence=0.9

directive는 line 시작이 정확히 @fact 여야 합니다. code fence나 HTML comment 안의 directive는 무시됩니다.

허용 relation과 option

PREFERS, DISLIKES, USES, WORKS_AT, HAS_PROJECT, PART_OF, SUPERSEDES, MENTIONS

valid_from
유효 시작 시각
valid_to
유효 종료 시각. valid_from보다 뒤여야 합니다.
learned_at
학습 시각. 생략하면 memory의 updated_at을 사용합니다.
expired_at
명시적 만료 시각. validity time format과 같아야 합니다.
confidence
0.0부터 1.0까지의 숫자
object_kind
entity (기본) 또는 value
시간 형식: signed 64-bit Unix seconds (예: -1), zero-padded YYYY-MM-DD, 또는 UTC YYYY-MM-DDTHH:MM:SSZ. valid_from, valid_to, expired_at은 한 fact 안에서 같은 형식을 써야 합니다.

Fact query

# 기본: 현재 유효한 active-memory facts
rem facts

rem facts --entity User
rem facts --relation PREFERS
rem facts --at 2025-04-01
rem facts --at -1
rem facts --all
rem facts --entity User --source

Filter semantics

--entity는 subject 또는 object의 canonical name을 case-insensitive하게 match합니다. --relation은 공백·hyphen을 underscore로 normalize한 후 대문자 relation과 비교합니다.

Output semantics

기본 출력은 fact ID, subject, relation, object, valid_from, valid_to, expired_at, learned_at, confidence, source memory ID의 TSV입니다. --source는 source path, episode ID, excerpt를 대신 포함합니다. --at과 --all은 함께 쓸 수 없습니다.

08 · health and recovery

Index rebuild와 doctor

SQLite는 local cache이므로 필요할 때 재생성할 수 있습니다. rebuild는 Git commit을 만들지 않고 index만 temporary file에서 atomic replace합니다.

Cache refresh

rem rebuild
rem search --bm25 "SQLite"
rem facts --entity User

rebuild output은 indexed memory 수, diagnostics, semantic entity/episode/fact 수를 보여줍니다. malformed Markdown, duplicate memory ID, invalid @fact는 diagnostic이 됩니다.

전체 health check

rem doctor

vault directory, default policy 파일, index/diagnostics, semantic cache schema, Git remote, ignore rules, pending transaction journal을 ok / warn 형태로 출력합니다.

남은 transaction journal: crash 등으로 <vault>/.rem/tx/<txid>가 남으면 doctor가 warning을 냅니다. vault를 확인·복구한 뒤 rem commit을 다시 실행하세요. cache와 tx directory는 Git에 commit하지 않습니다.

09 · scripting

Agent와 CI에서 안전하게 실행하기

script는 prompt에 의존하면 안 됩니다. global config를 격리하고 external-change policy를 명시하세요.

export REM_HOME=/tmp/rem-agent-home

# profile이 아직 없으면 init이 profile도 저장한다.
rem init \
  --root /workspace/vault \
  --storage git \
  --non-interactive \
  --accept-external

rem add --short \
  --scope agent \
  --source agent \
  --non-interactive \
  --accept-external \
  "Build passed; release candidate is ready."

# agent가 직접 편집한 Markdown을 함께 반영
rem commit --non-interactive --accept-external -m "sync agent memory"

전제조건 checklist

  • profile root가 Git top-level directory다.
  • origin이 github.com 또는 gitlab.com이다.
  • index/reindex가 valid Markdown과 semantic fact를 만들 수 있다.
  • external Git changes 처리 방침을 명시했다.

자동화에서 피할 것

  • --non-interactive와 policy flag 없이 dirty vault를 mutation하지 않습니다.
  • --restore-external은 manual/untracked 변경을 버릴 수 있으므로 자동 기본값으로 두지 않습니다.
  • Git unmerged conflict를 rem command로 억지로 해결하려 하지 않습니다.
  • --vector는 현재 v1에서 구현되어 있지 않습니다.

10 · complete command reference

전체 CLI 명령과 옵션

아래는 rem --help와 모든 subcommand help를 기준으로 정리한 전체 surface입니다. 전역 option은 -h, --help와 -V, --version입니다.

Configure, profile, config, vault

명령사용법 / 옵션설명
rem인자 없음rem configure와 같은 TUI를 엽니다.
rem configure옵션 없음Interactive configuration TUI.
rem init--root <path> --storage local|obsidian|git --profile <name> + mutation optionsvault directory/policy/cache를 만들고 profile을 저장하며 initialize Git commit을 만듭니다.
rem profile list옵션 없음모든 profile 및 active marker를 출력합니다.
rem profile show[name]active 또는 지정 profile의 name/storage/root를 출력합니다.
rem profile add<name> <root> --storage local|obsidian|git이미 유효한 Git vault를 profile에 add/replace합니다.
rem profile use<name>active profile을 변경합니다.
rem config init|path|show|reset옵션 없음config 생성, path 출력, TOML 출력, default reset.
rem config setactive-profile|default-search|editor <value>한 config value를 갱신합니다.

Memory lifecycle

명령사용법 / 옵션설명
rem add[TEXT...]; --short|--long|--type short|long; --scope user|project|agent|session; --kind fact|preference|decision|task|procedure|note|question; repeated --tag; --source; --agent; --session + mutation options새 active memory를 만들고 자동 commit합니다.
rem list--short|--long|--type; --scope; --kind; --tag; --allfilter된 TSV memory 목록입니다.
rem show<id-or-unique-prefix>memory Markdown frontmatter와 body를 출력합니다.
rem edit<id> + mutation options외부 editor로 열고 성공 종료 시 reindex/commit합니다.
rem update<id> [TEXT...]; --append + mutation options본문을 replace하거나 append합니다.
rem delete<id>; --hard + mutation options기본 archive, --hard는 파일 삭제입니다.
rem promote<short-id> [TEXT...] + mutation options새 long memory를 만들고 promoted_from을 기록합니다.

Transaction, retrieval, health

명령사용법 / 옵션설명
rem commit-m, --message <text>; --accept-external; --restore-external; --review; --dry-run; --non-interactivepending Markdown 변경을 validate, reindex, Git commit합니다.
rem search[QUERY...]; --grep; --bm25; --vector; --all; --mode auto|grep|bm25|vector|allactive memory 검색입니다.
rem facts--entity <name>; --relation <relation>; --at <time>; --all; --sourcederived semantic facts TSV를 query합니다. --at과 --all은 배타적입니다.
rem rebuild옵션 없음SQLite search/semantic cache만 재생성합니다. Git commit 없음.
rem doctor옵션 없음config, vault, policy, cache, Git, journal 상태를 검사합니다.
Mutation option summary: --accept-external은 기존 vault 변경을 현재 commit에 포함하고, --restore-external은 그 변경을 복원/폐기하며, --non-interactive는 prompt를 막습니다. 두 policy flag는 함께 쓸 수 없습니다. rem commit --review은 두 policy flag 및 --non-interactive와 함께 쓸 수 없습니다.

11 · recovery guide

자주 마주치는 오류와 대응

증상원인대응
rem: command not foundCargo user bin directory가 PATH에 없음export PATH="$HOME/.cargo/bin:$PATH"를 shell profile에 추가하고 새 terminal을 엽니다.
vault root is not a Git repository / must be the Git repository rootroot가 Git repository가 아니거나 subdirectory임git -C <root> rev-parse --show-toplevel이 root와 같은지 확인합니다.
remote origin must point to GitHub or GitLaborigin 없음 또는 지원되지 않는 hostgit -C <root> remote add origin git@github.com:you/vault.git처럼 설정합니다.
external vault changes detected기존 manual / Obsidian / agent 변경이 있음interactive면 선택하고, script면 --accept-external 또는 --restore-external을 명시합니다.
unmerged Git conflict detectedGit merge/rebase conflict가 아직 해소되지 않음Git/editor로 conflict marker를 해결하고 Git conflict를 끝낸 뒤 rem commit --review를 다시 실행합니다.
reindex produced diagnosticsmalformed frontmatter, duplicate ID, invalid semantic fact, symlinked memory 등오류가 가리킨 Markdown을 고친 뒤 다시 실행합니다. transaction mutation은 rollback됩니다.
BM25/facts index does not existSQLite cache가 없음 또는 오래됨rem rebuild를 실행합니다.
semantic cache schema missing이전 index 또는 cache 손상rem rebuild로 local cache를 재생성합니다.
pending transaction journal이전 process가 crash 또는 강제 종료rem doctor 결과를 확인하고 vault recovery 후 rem commit을 재시도합니다.
최소 진단 순서: rem profile show → rem doctor → git -C <vault> status --short → 필요 시 rem rebuild → rem commit --dry-run.
이 문서는 repository의 rem --help, 모든 subcommand help, README.md, docs/local-deployment.md, 현재 구현을 기준으로 작성했습니다. Markdown과 Git history가 원본이며 SQLite cache는 재생성 가능합니다.
